Purpose

To provide a standardized process for diagnosing, escalating, and resolving network and connectivity issues across infrastructure, ensuring minimal downtime and clear communication with stakeholders. It serves as guidance for L1 IT support in handling network switch-related issues that impact store and office connectivity.

When to Use This Runbook?

A. User-Reported Issues
• Stakeholder cannot connect to the internet, Wi-Fi, or VPN such as PDTs, RM’s laptop, HR Kiosks, SM CIS Kiosks and CTD mobile phones
• BDO Remittance, CTD web applications, MSPG, POS and EFT terminals fail to process transactions due to connectivity errors.
• Stakeholders report unstable network connections.

B. Monitoring Alerts
• Network monitoring tools detect outages, packet loss, or unusual latency.
• Bandwidth utilization spikes beyond thresholds, affecting business-critical applications.

C. Site-Wide or Multi-Department Impact
• Entire store, office, or branch loses connectivity.
• Multiple departments report simultaneous network issues.
• Business-critical services (All MSPG terminals application, all CTD Web applications, BDO remittance, EFTs bank hosts) are inaccessible.

D. Hardware or ISP Failures
• Suspected router, switch, or access point malfunction.
• ISP outage confirmed or suspected.
• Hardware replacement or vendor intervention required.

E. Escalation Scenarios
• First-line troubleshooting exceeds 30 minutes without resolution.
• Issue impacts revenue, compliance, or customer experience.
• RCA of the incident at the level of NOC (Specialized, data ownership, authority).

Prerequisites

• Documentation (reference ticket number) and communication channels in place.

• Access to workstations and systems
o Basic Network Health Checks
- Ability to confirm if local switches, routers, and firewalls are up and reachable.
- Visibility into device status (CPU, memory, interface utilization).
- Ability to detect if issues are localized (branch-only) or widespread.

• Monitoring visibility
o Connectivity Verification
- Tools to test WAN/LAN connectivity (ping, traceroute, bandwidth tests).
- Ability to confirm if branch circuits (ISP links) are up or down.

• Local Infrastructure Monitoring
- Ability to detect if issues are localized (branch-only) or widespread.

• Escalation Readiness

A. Escalation contacts ready
- Knowledge of escalation matrix (who to contact, in what order).
- Preparedness to provide evidence (screenshots, logs, test results) to NOC.

B. Incident Documentation
- Clear capture of symptoms (e.g., latency, packet drops, disconnections).
- Timestamping when the issue started and when it was observed.

C. Broadcast as needed.

Step-by-Step Procedure

A. Detection & Logging
• Receive issue report from end-user, helpdesk, or Manage Engine.
• Record incident details in the resolution on ME (time, user, symptoms, affected services).
• Confirm if the issue is isolated (single device) or widespread (multiple users/branch).

B. Basic Checks
• ✅ Verify physical connections: patch cables, switches, Wi-Fi access points, network UPS.
• ✅ Check device network settings: IP address, DNS, DHCP status.
• ✅ Restart affected device (PC, POS terminal, router/AP if permitted).
• ✅ Ping default gateway and DNS server to confirm local connectivity.

C. Quick Troubleshooting
• Test connectivity to internal systems (intranet sites).
• Test external connectivity (internet sites).
• Check if issue is user-specific (credentials, device settings) or network-wide.
• Document all steps taken in the incident log.

D. Communication
• Inform branch manager or designated contact about the issue.
• Provide updates every 30–60 minutes until resolution or escalation.
• Share incident ticket reference number for tracking.

E. Resolution & Closure
• If resolved at branch level:
o Confirm with affected users that services are restored.
o Document resolution steps in the ticket.
• If escalated:
o Hand over incident details, troubleshooting steps, and logs to Network Group.
o Close ticket at branch level once higher support confirms resolution.

F. Preventive Follow-Up
• Note recurring issues for pattern analysis.

Communication Guidelines

A. Tone & Style
• ✅ Be professional, polite, and concise.
• ✅ Use clear, non-technical language when speaking to non-IT staff.
• ✅ Avoid jargon unless communicating with technical teams.
• ✅ Stay calm and reassuring, especially during outages.

B. Information to Include in Updates
Every communication should cover:
• Incident Reference Number (ticket ID).
• Issue Summary: What is affected (POS, MSPG, EFT, Wi-Fi, VPN, etc.).
• Scope: Single user, department, or branch-wide.
• Actions Taken: Steps already performed at branch level.
• Next Steps: Escalation or pending resolution.
• ETA/Update Frequency: When stakeholders can expect the next update.

C. Communication Channels
• Internal Users (Branch Staff/Managers):
o Use email, chat, or phone for quick updates.
o Keep explanations simple and focused on impact.
• Network Group / 2nd Level Support:
o Use incident ticketing system or escalation email.
o Provide detailed technical logs and troubleshooting steps.
• Vendors/ISPs:
o Contact via official support channels.
o Share precise error details, device IDs, and timestamps.

D. Timing & Frequency
• Initial Notification: Immediately after confirming an incident.
• Regular Updates: Every 30–60 minutes until resolution or escalation.
• Closure Notice: Once issue is resolved or handed over to higher support.

E. Escalation Communication Template
Subject: [Branch] Network Connectivity Issue – Escalation Required Body:
• Incident ID: [Ticket Number]
• Branch: [Location]
• Issue: [Summary of problem]
• Scope: [Number of users/systems affected]
• Steps Taken: [List of troubleshooting actions performed]
• Request: Escalation to Network Group for advanced support
• Contact: [Branch IT name & phone/email]

F. Do’s & Don’ts
• ✅ Do acknowledge user concerns promptly.
• ✅ Do document all communications in the incident log.
• ❌ Don’t speculate on causes without evidence.
• ❌ Don’t promise resolution times unless confirmed by higher support.
• ❌ Don’t use informal or emotional language in official updates.

Validation / Closure

A. Validation Before Closure
• Confirm Service Restoration
o Test connectivity (ex. internet browsing, VPN access, POS/MSPG/EFT transaction, web applications).
o Verify affected devices are back online (PCs, POS terminals, APs).
o Ensure branch staff confirm that their operations are functioning normally.

• Confirm Scope
o Ensure the issue is resolved for all affected users, not just one device.
o Confirm no residual impact on other services.

B. Documentation
• Record the following in the incident ticket:
o Timeline of events (detection, troubleshooting, escalation, resolution).
o Actions taken by branch IT and higher-level teams.
o Root cause (if identified).
o Preventive measures or recommendations.

C. Communication
• Send closure notification to branch manager and stakeholders:
o Incident ID and summary.
o Confirmation that services are restored.
o Root cause (if known) and resolution steps.
o Preventive actions (e.g., firmware update, ISP failover test).

• Update Network Group / 2nd Level Support:
o Provide final status and confirm no further action required.
o Share logs or monitoring screenshots if requested.

D. Closure Criteria
Branch IT can close the incident when:
• ✅ All affected services are validated as operational.
• ✅ Stakeholders confirm normal operations.
• ✅ Documentation is complete in the ticketing system.
• ✅ Escalation teams (if involved) confirm resolution.

E. Post-Closure Follow-Up
• Note recurring issues for pattern analysis.
• Update branch IT knowledge base with lessons learned.
• Schedule preventive checks.

Documentation Requirements

A. Incident Identification
• Ticket Number (Manage Engine)
• Date & Time of detection and reporting.
• Branch Location and affected department(s).
• Reporter Details (user name, contact, role).

B. Issue Description
• Symptoms: What users experienced (e.g., “POS terminals offline status”).
• Scope: Single device, multiple users, or branch-wide.
• Affected Services: Internet, VPN, MSPG, POS, email, etc.
• Initial Impact Assessment: Business disruption, revenue impact, or operational delay.

C. Troubleshooting Actions
• Basic Checks Performed: cables, IP/DNS, gateway ping, device restart.
• Intermediate Steps: traceroute, firewall/VPN validation, router/switch log review (c/o NOC)
• Results of Each Step: success/failure, observations.
• Time Spent: duration of troubleshooting before escalation.

D. Escalation Details
• Escalation Trigger: reason for escalation (e.g., unresolved after 30 minutes, critical service outage).
• Escalation Contact: Network Group, CFO, L2
• Information Shared: logs, screenshots, error messages, troubleshooting summary.

E. Communication Log
• Notifications Sent: branch manager, stakeholders, higher support.
• Update Frequency: timestamps of each communication.
• Closure Confirmation: acknowledgment from stakeholders that services are restored.

F. Resolution & Validation
• Resolution Method: config change, hardware replacement, ISP restoration.
• Validation Tests: POS transaction, VPN access, internet browsing.
• User Confirmation: feedback from affected staff confirming normal operations.

G. Closure Notes
• Root Cause (if identified): hardware fault, ISP outage, misconfiguration.
• Preventive Measures: firmware update, failover testing, monitoring adjustment.
• Knowledge Base Update: add new troubleshooting steps or lessons learned.
